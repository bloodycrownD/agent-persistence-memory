# 知识库（kb）与 .apm 目录重组 PRD

## 背景

当前 APM CLI 将角色、持久记忆、动态明细等落在 `.apm` 顶层或浅层子目录；知识内容无独立「知识库」入口，检索能力弱。需要在**同一工作区模型**下引入可维护的知识树、可重建的检索索引，并把「动态明细」与「持久记忆」在目录上收拢到 `memory/`，与知识库 `kb/` 并列，便于人机协作与后续自动化。

## 目标（含成功指标）

1. **可初始化**：用户在空目录或新仓库能通过一条命令得到完整、可理解的 `.apm` 骨架（见范围）。
2. **知识可进可搜**：Markdown 知识文件可导入或写入到约定树内；用户能用关键词检索到相关片段或文档，且结果可解释（至少含标题/路径/摘要或片段）。
3. **动态可收尾**：动态区支持归档与清空，行为与「阶段结束」心智一致，避免长期堆在单文件里难以审计。
4. **成功指标（首版可测）**
   - 在标准夹具目录上执行 `init` 后，目录结构符合「包含范围」的树形约定，且无手改即可通过 `help`/自检清单。
   - 同夹具上完成「导入 → 搜索」闭环：对已知关键词的查询，**Top 结果中至少一条**命中预期文档（可写在夹具说明中）。
   - `dynamic archive` 执行后，归档目录中出现**带时间戳**的归档产物，且主动态区状态符合验收描述；`dynamic clear` 行为与验收一致。

## 用户与场景

| 用户 | 场景 |
|------|------|
| 使用 APM 的开发者 / Agent 操作者 | 初始化工作区，建立 role、persist、dynamic 与 kb 的约定位置。 |
| 维护长期规则与笔记的人 | 将已有 Markdown 知识树导入 kb，按关键词检索复习或引用。 |
| 推进阶段性任务的人 | 在 dynamic 中记录进展；阶段结束时 **archive** 留痕，或 **clear** 重新开始。 |

## 范围

### 包含范围

- **CLI**
  - 新增顶层子命令 **`kb`**：具备**写入**、**导入**、**搜索**等能力（具体子命令命名以实现为准，但能力需齐备）。
  - **`dynamic`**：与 **`persist`** 在「读写的直观性」上对齐——以**少而清晰**的子命令完成读、写、改、归档、清空（与当前「仅 detail + show」相比简化/对齐 persist 的体验目标）。
  - 新增 **`init`**：创建本 PRD 约定的 `.apm` 下目录与占位/默认文件（含 `config`/`status` 等若产品仍需要，以保持 CLI 可运行）。
- **目录布局（均在 `.apm` 下，破坏性重组）**
  - **`.apm/kb/`**：至少包含 **`docs/`**（可嵌套子目录，**不打平**）、**`dynamic/`**（知识库侧过程笔记，如 `detail.md`；与 memory 侧任务动态**路径隔离**）。
  - **`.apm/memory/`**：**扁平主文件** + 归档目录——**`role.md`**、**`persist.md`**、**`dynamic.md`**（分别承载角色、持久记忆、任务动态正文）。
  - **`.apm/kb/`**：至少包含 **`docs/`**（可嵌套子目录，**不打平**）、**`dynamic/`**（知识库侧过程笔记，如 `detail.md`；与 memory 侧任务动态**路径隔离**）、**`archive/`**（仅承接 `dynamic` 的归档副本，时间戳命名）。
- **知识库检索（产品能力描述）**
  - 支持**倒排或等价全文索引**能力，并对中文场景有可用的**分词或等价切分策略**（效果上接近「能搜到词/短语」，不要求与某一商业搜索引擎一致）。
  - 索引持久化：**JSON 形态且 gzip 压缩**落盘；路径与文件名以实现为准，须写入 `.apm/kb/` 下合理位置且不污染 `docs/` 源文。
- **`dynamic` 归档**
  - **`archive`**：将当前 **`memory/dynamic.md`** 全文归档到 **`.apm/kb/archive/`** 下**带时间戳**的文件（或等价单文件命名），主动态区后续状态在验收中定义。
  - **`clear`**：清空或重置 **`memory/dynamic.md`** 为「空工作态」，**不**静默删除 **`.apm/kb/archive/`** 内已归档文件。

### 不包含范围

- **旧路径兼容**：不保留对历史 `.apm/role.md`、`.apm/persistence/`、`.apm/dynamic/detail.md` 等布局的读取或自动迁移（与「破坏性重组」一致；升级说明可在发布层一句带过，**不在本 PRD 写迁移脚本需求**）。
- **非 Markdown 知识导入**（如 PDF、Office）：首版不包含；若 CLI 暴露导入参数，仅面向 **Markdown（.md）** 树。
- **远程多机同步、权限模型、多用户协作**：不包含。
- **具体 npm 包名、索引 JSON Schema、类图与 API 列表**：属技术方案，**不在本 PRD 展开**。

## 核心需求（3–7 条）

1. **`apm init`**：在 `cwd` 下创建 `.apm` 及 **`kb/`**、**`memory/`** 约定结构（含 `kb/docs`、`kb/dynamic`、`memory/role.md`、`memory/persist.md`、`memory/dynamic.md`、`memory/archive/` 等约定节点）；与现有 `config`/`lock` 等机制兼容所需文件一并创建。
2. **`apm kb`**：支持向知识树**写入**单篇或批量内容、从指定路径**导入** Markdown 树、对知识库执行**关键词/短语搜索**并输出人类可读结果列表（含定位信息）。
3. **知识树与索引**：`kb/docs` 下允许**任意深度子目录**；索引文件为 **gzip 压缩的 JSON**；重建索引的命令或流程在实现中明确并在验收中可测。
4. **`apm dynamic` 对齐 persist 心智**：子命令集**简化**，覆盖「读当前内容、覆盖写、区间编辑（若与 persist 一致则保留）、**archive**、**clear**」；与 `kb` 的 dynamic 在路径上严格区分。
5. **`dynamic archive`**：生成**时间戳命名**的归档副本于 **`.apm/memory/archive/`**；不覆盖同名历史归档（除非实现采用唯一名策略并在验收中说明）。
6. **`dynamic clear`**：将 **`memory/dynamic.md`** 恢复为约定空态；**不得**删除 **`memory/archive/`** 内已有归档（除非用户显式另有命令，本需求不包含）。

## 验收标准

### init

- **Given** 空目录且无 `.apm`  
- **When** 用户执行 `apm init`（或文档化后的等价入口）  
- **Then** 出现 `.apm/kb/docs`、`.apm/kb/dynamic`、`.apm/kb/archive`、`.apm/memory/role.md`、`.apm/memory/persist.md`、`.apm/memory/dynamic.md`，且后续 `apm kb --help` / `apm dynamic --help` 可正常输出。

### kb 导入与搜索

- **Given** 标准夹具：`.apm` 已由 init 创建，`kb/docs` 下存在至少两篇不同关键词的 `.md`  
- **When** 用户执行导入（若导入为独立子命令）或写入已完成，并执行搜索命令且查询词为夹具中某关键词  
- **Then** 输出非空结果列表，且 **Top N（如 N=5）中至少一条** 命中预期文档；结果中含可定位字段（如相对 `kb/docs` 的路径或标题）。

### 索引文件

- **Given** 已完成至少一次索引构建  
- **Then** 在 `.apm/kb/` 下存在 **gzip 压缩**的 JSON 索引产物；删除该文件后，产品文档或 `--help` 中说明的「重建索引」操作可恢复搜索能力（验收可通过脚本或单测断言文件扩展名与 magic gzip）。

### dynamic archive / clear

- **Given** **`memory/dynamic.md`** 主内容非空  
- **When** 执行 `dynamic archive`  
- **Then** `.apm/kb/archive/` 下新增**带时间戳**的文件，且内容与归档前主内容一致（或实现定义的等价）；主区行为在实现说明中二选一并在验收中固定：**保留空 front matter 模板**或**截断为约定空文**，须与 `clear` 语义不冲突。

- **Given** `.apm/kb/archive/` 下已存在至少一个归档文件  
- **When** 执行 `dynamic clear`  
- **Then** 主动态区为约定空态；**`.apm/kb/archive/` 内文件数量不减少**（除非 clear 文档明确误用，本需求以不删归档为准）。

### 破坏性

- **Given** 仅含**旧版** `.apm` 布局的工作区  
- **When** 用户仅升级 CLI、不手工迁移  
- **Then** 不要求 CLI 自动迁移；但 `init` 在新目录上行为符合上文（旧区兼容性非本 PRD 验收项）。

## 扩展章节：风险与待确认项

- **技术选型**（倒排实现、中文分词库）由实现阶段确定，本 PRD 只约束**可测效果**与索引 gzip+json 形态。
- **`kb/dynamic` 与 `memory/dynamic.md` 的边界**：须在实现/用户文档中一句话区分——**知识库侧过程笔记**（`kb/dynamic/…`）与**任务/会话动态**（`memory/dynamic.md`）。
- **已定产品形态**：`memory` 下 persist/role/dynamic 采用 **同级单文件**（`persist.md` / `role.md` / `dynamic.md`），与 SPEC 一致；若未来 persist 需多文件，再开迭代拆子目录。
