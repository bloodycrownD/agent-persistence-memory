# apm read 联想区 PRD

## 背景

当前 `apm read` 仅聚合输出 **角色**、**持久记忆**、**动态记忆** 三段正文，Agent 初始化时无法从命令输出中直接看到与当前上下文相关的知识库命中。

项目已具备基于 MiniSearch 的知识库检索能力（倒排索引 + BM25+），但检索入口独立在 `apm kb search`，未融入 `apm read` 工作流。需要在 `apm read` 中增加 **联想区**，在动态记忆之后展示与当前记忆上下文最相关的知识库条目，并分层展示详略。

## 目标（含成功指标）

1. **一体化初始化**：Agent 执行一次 `apm read` 即可同时获得三段记忆 + 知识库联想结果，无需再单独调用 `apm kb search`。
2. **可解释的相关性**：详细区结果展示匹配率、路径、命中关键词及原文行，便于人工快速判断是否要打开文件。
3. **可测、可判定**：在固定夹具下，联想区条目数、排序、格式满足本文验收标准；`npm test` 可覆盖核心场景。

**成功指标**

| 指标 | 目标 |
|------|------|
| 命令入口 | `apm read` 文本输出在 **# 动态记忆** 之后出现 **# 联想区**（有命中时） |
| 详细区容量 | 最多 **5** 条，每条最多 **3** 行非空匹配正文 |
| 简略区容量 | 在详细区之后，最多再展示 **10** 条（与详细区不重复文件） |
| 检索范围 | `.apm/kb/` 下除 `index/` 外全部 Markdown 文件均可被联想命中 |
| 本次范围外 | 不实现 `apm read --json` |

## 用户与场景

| 用户 | 场景 |
|------|------|
| 外部 Agent | 会话恢复或任务启动时执行 `apm read`，根据联想区快速定位可能相关的 kb 文档 |
| 开发者 | 本地调试记忆与知识库是否对齐，肉眼检查匹配率与命中行是否合理 |

## 范围

### 包含范围

1. **输出位置**：`apm read` 文本中，**# 动态记忆** 区块之后输出 **# 联想区**。
2. **查询上下文**：将 **role + persist + dynamic** 三段正文（去除 front matter 后的有效正文）合并，作为本次联想的检索上下文。
3. **检索对象**：`.apm/kb/` 目录下 **除 `index/` 以外** 的全部 Markdown 文件，包括但不限于：
   - `kb/docs/**`
   - `kb/dynamic/**`
   - `kb/archive/**`
4. **匹配率**：展示为 **0–100 的整数百分比**（如 `[87%]`），在 **当次 `apm read` 参与排序的全部候选结果** 内对 BM25+ 原始分数做归一化（最高分映射为 100%，其余按比例取整）。
5. **关键词**：展示与本次查询上下文 **实际命中** 的检索词；由与知识库索引一致的分词逻辑产生，并过滤介词、email、标点等噪声（与索引侧停用/过滤规则一致，保证可解释）。
6. **详细区（最多 5 条）**：按相关度降序，每条包含：
   - 首行：`[匹配率] <文件路径> <关键词>`（关键词为命中词，多个时以空格或约定分隔符展示，需在实现中固定一种格式并在测试中锁定）
   - 其后：该文件中 **包含命中检索词的非空行**，最多 **3** 行；每行格式为 `行号|正文`（行号为源文件中的 1-based 行号，正文为该行完整内容，不截断）
   - 详细区条目之间空一行分隔
7. **简略区（最多 10 条）**：接续详细区之后、与详细区 **不重复同一文件** 的下一批结果；每条仅一行：`[匹配率] <文件路径> <关键词>`，**不**展示匹配行正文。
8. **数量策略（固定上限、不补位）**：始终 **尝试** 取满 **5** 条详细 + **10** 条简略；候选不足时 **有几条展示几条**，**不** 用空行或占位符补齐。
9. **索引依赖**：联想依赖知识库检索索引；索引缺失或需重建时的表现纳入验收（见下）。

### 不包含范围

- `apm read --json` 及任何结构化字段扩展
- **持久化关联**、**联想关键词** 等历史 V1 模板区块（与本次「知识库文件联想」为不同能力）
- 调用外部 LLM 生成摘要、关键词或改写查询
- 修改 `apm kb search` CLI 的独立行为（可复用同一套检索逻辑，但不在本 PRD 内规定 CLI 变更）
- 用户自定义停用词表、匹配率算法、分词器配置的 UI 或 `config.json` 项（使用项目内统一规则即可）

## 核心需求

1. **`apm read` 增加联想区区块**：在动态记忆之后，按模板输出 `# 联想区` 及分层内容。
2. **查询上下文合并**：role、persist、dynamic 正文合并为单一查询文本；任一段为空时仍用其余段参与检索。
3. **检索范围覆盖 kb 全量 Markdown（除 index）**：索引构建与检索范围须与 PRD 一致，避免仅 `kb/docs` 可搜而 `dynamic`/`archive` 不可搜。
4. **相关度排序与分层**：全库检索后按 BM25+ 分数降序；前 5 个 **不重复路径** 的文件进入详细区并附命中行；其后最多 10 个 **未出现在详细区** 的文件进入简略区。
5. **匹配率展示**：详细区与简略区首行均带 `[NN%]`，NN 为 0–100 整数，当次结果集内归一化。
6. **关键词可解释**：关键词来自「查询分词结果 ∩ 文档命中词」类规则，经停用词/噪声过滤，不得展示与本次上下文无关的整文档 keywords 元数据冒充命中词。
7. **详细区命中行**：仅非空行；行内须包含至少一个命中检索词；最多 3 行；格式 `行号|正文`。

## 验收标准

### 输出结构与位置

- [ ] **AC-1**：当 role、persist、dynamic 至少一段有内容且知识库存在 ≥1 条命中时，`apm read` 输出顺序为：`# 角色` → `# 持久记忆` → `# 动态记忆` → `# 联想区`（前三个区块仍遵循「无内容则省略该 `#` 区块」的现有规则）。
- [ ] **AC-2**：`# 联想区` 下第一条详细结果的首行符合模式：`[<整数>%] <路径> <关键词...>`，且路径以 `.apm/kb/` 下相对路径或项目约定的 kb 内相对路径展示（实现选定一种并在测试中固定）。

### 详细区（最多 5 条）

- [ ] **AC-3**：在夹具中准备 ≥8 个可被同一查询命中的 kb 文件时，详细区展示 **5** 条，且按匹配率降序排列。
- [ ] **AC-4**：每条详细结果下方有 **1–3** 行 `行号|正文`，对应源文件非空行且该行包含至少一个展示在首行关键词中的命中词；**不出现**空 `行号|` 或仅空白正文行。
- [ ] **AC-5**：在仅 2 个文件可命中时，详细区只展示 **2** 条，**不** 以空行凑满 5 条。

### 简略区（最多 10 条）

- [ ] **AC-6**：在夹具中准备 ≥20 个可命中文件时，详细区 5 条 + 简略区 **10** 条，共 **15** 个 **不重复** 文件路径。
- [ ] **AC-7**：简略区每条 **仅有** 一行 `[匹配率] 路径 关键词`，**无** `行号|正文` 子行。
- [ ] **AC-8**：简略区与详细区 **无路径重复**；若总命中数 ≤5，则 **无** 简略区（或简略区 0 条）。

### 检索范围与上下文

- [ ] **AC-9**：在 `kb/dynamic/detail.md` 或 `kb/archive/` 下放置含特征词的 Markdown（且 `kb/docs` 中无该词）时，合并 role+persist+dynamic 上下文检索后，该文件可出现在联想区（证明范围覆盖 index 以外 kb 文件）。
- [ ] **AC-10**：修改 role 正文中特征词后再次 `apm read`，联想区排序或命中集合随之变化（证明查询上下文含 role，而非仅 dynamic）。

### 匹配率与关键词

- [ ] **AC-11**：同一次 `apm read` 中，详细区与简略区所有 `[NN%]` 满足 `0 ≤ NN ≤ 100`，且至少一条为 **100**（当存在非零分数时）。
- [ ] **AC-12**：首行关键词中 **不包含** 明显停用/噪声 token（实现需提供固定停用表；测试中至少断言常见英文介词 `the`/`a` 与纯标点不会作为唯一关键词展示）。

### 索引与空结果

- [ ] **AC-13**：当 `.apm/kb/index/search.json.gz` 不存在时，`apm read` **仍正常输出** 角色/持久/动态；联想区输出 **明确提示** 需执行索引重建（文案需包含 `kb index rebuild` 或等价指引），**不** 导致命令非零退出。
- [ ] **AC-14**：当索引存在但 **无任何命中** 时，**不输出** `# 联想区` 整段（与「固定上限、不补位、无结果不占位」一致）。

### 回归

- [ ] **AC-15**：现有 `apm read` 三段记忆行为与测试用例保持通过；新增 Vitest 用例覆盖 AC-3、AC-6、AC-9、AC-13 中至少各一条。
