# read-display-tiers 变更需求说明

## 变更动机与原因

在 `read-keyword-association` 等实现之后，产品层仍与**最初 prompt 模板**在「展示形态」上存在可感 gap。经确认：

- **联想记忆**不需要再对单条 chunk 做「物理切段」：在 **`apm chunks add` 起即限制单条 `text`（正文）≤ 200 字** 时，**前 3 条完整展示即等价于 3 段「联想片段」**（每段 ≤200 字）。
- **持久化关联**与**联想区**均采用**同一类交互**：先按**倒排/词频式**得到与上下文相关的匹配度排序，再按 **3（完整）+ 5（摘要）** 两档输出；其中「完整」指输出 chunk **正文全量**（在 ≤200 约束下即整段），「摘要」指仅 **name + keywords**。
- **Todos** 不采用「5～10 条」软区间，改为 **硬上限 20 条**（超过视为分步过细或需合并任务）。

## 与主需求 / 前序 feature 相比的变化

| 维度 | 前序 `read-keyword-association` / 当前实现习惯 | 本变更目标 |
|------|----------------------------------|------------|
| 持久化关联下的 chunk 展示 | 最多 5 条，且列表多为 name+keywords | 匹配度排序后 **前 3 条输出完整正文**；**第 4～8 条**仅 **name + keywords**（无正文） |
| 联想区结构 | 与全量 `## Chunks` 易混淆；「联想记忆」未独立成节 | 增加/明确 **## 联想记忆**；其下为 **3 完整 + 5 摘要**；**## 联想关键词** 单独一行（5～10 个词，规则见下） |
| 联想区排序源 | 与持久化关联可能混用同一队列 | **独立**一套「联想区」匹配度：更侧重 **tmp detail + 当前未完成任务（首条）**；**可**叠加热门 persist 词，但需在实现中写清权重（见 plan） |
| Chunk 正文长度 | 主 spec 未写死；实现可能无校验 | **`chunks add` / `chunks edit` 写入的 text** 使用与主 spec 一致的字数统计，**正文 ≤ 200 字**（与「每段 ≤200」对齐） |
| Todos 条数 | 无硬上限 | **全库 todo 条数 ≤ 20**；`todos add` 在将突破上限时 **失败** 并提示 |
| 当前任务长描述 | 与「500～1000 自动生成」字面易冲突 | **明确为 spec 对齐**：**当前任务**由 **首个未完成 todo**（按现有优先级规则）的 `name` + `description` 直接拼出；**长文执行记录**以 **tmp detail 正文** 为准（其本身受 500～1000 字限制），**不**再要求单独生成第二段 500～1000 字叙述 |

## 影响模块与接口

- **`apm read` / `apm read --json`**：模板章节顺序与字段需反映「持久化关联（分层）」「联想记忆（分层）」「联想关键词」；JSON 需可被 Agent 稳定解析（字段名在 plan 中固定）。
- **`apm chunks add|edit`**：增加 **正文 ≤200 字** 校验。
- **`apm tmp todos add`（及必要时 edit/import）**：增加 **全局 todo 数量 ≤20** 校验。
- **Vitest**：覆盖分层输出字数边界、todos 上限、chunk 字数拒绝路径。

## 功能需求（增量）

### 1. Chunk 正文上限（支撑「片段 ≤200」）

- `apm chunks add --text`：`countChars(text) <= 200`，否则报错提示。
- `apm chunks edit --text`：同上（若未传 `--text` 则不改写，不触发）。

### 2. Todos 上限

- 任意时刻 `.apm/tmp/todos/*.md` 条目数 **≤ 20**。
- `tmp todos add`：若当前已有 20 条，**拒绝添加**并提示需合并/清理/归档。

### 3. `apm read` 文本模板（与最初 prompt 对齐）

固定包含以下逻辑块（与主 `spec` 不冲突的前提下补全展示）：

1. **## 持久化关联**（与「倒排查询词」一致，见下）
   - 首行：`Keywords: ...`（5～10 个，来自 persist + tmp detail + 未完成 todos 的抽取，规则延续前序 feature）。
   - **Primary（Top 3）**：对**按「持久化关联」匹配度排序**的 chunk 列表，取前 3 名，**逐条输出完整 `text`（正文）**（含 name/元信息行格式在 plan 中统一）。
   - **Secondary（Next 5）**：接上一排序结果，取**第 4～8 名**（若不足则少显示），**每条仅 name + keywords**。

2. **## 联想记忆**
   - 对**按「联想区」匹配度排序**的 chunk 列表（**独立打分队列**，与上一节排序可不同），同样 **Top 3 完整正文 + Next 5 仅 name+keywords**。
   - 若与「持久化关联」Top 出现重复 chunk：**允许重复展示**（两侧语境不同）；若实现选择去重，须在 plan 写明规则与测试。

3. **## 联想关键词**
   - 单行逗号分隔，**5～10** 个词；来源延续既定策略（例如由联想区 Top 命中 chunk 的 keywords 加权去重，排除与持久化 Keywords 完全重复项等——与实现一致即可，但必须固定为可测规则）。

4. **## Chunks（全量索引，可选）**
   - 是否保留「全量 chunk 列表」由实现决定；若保留，**必须不替代**上面两节的「分层」语义（可为附录性质）。

### 4. `apm read --json`（增量字段）

在现有字段基础上，增加能还原模板分层的最小结构（具体字段名以 plan 为准），至少区分：

- 持久化关联：**keywords**、**primaryChunks（含完整 content）**、**secondaryChunks（仅 name/keywords/score）**
- 联想记忆：同上结构（独立排序结果）
- **associativeKeywords**：与「## 联想关键词」一致

### 5. 当前任务（spec 对齐说明）

- **Current Task** 文本 = 首个未完成 todo 的 `name` 与 `description`（与现实现一致）。
- **不在此变更中**引入 LLM 生成额外「任务说明书段落」。

### 6. CLI 形态

- 维持现有 `chunks` 子命令 flag 设计（`--field` / `--q` / `--match` 等）；**不要求**恢复最初字面 `--keywords|--content|--name` 三选一。

## 验收标准

- [ ] `chunks add/edit` 在正文超过 200 字时失败且错误信息明确。
- [ ] `tmp todos add` 在第 21 条时失败。
- [ ] `apm read` 文本中 **持久化关联**、**联想记忆** 两节均体现 **3 完整 + 5 摘要**（在至少有 8 条可命中 chunk 的夹具下）。
- [ ] `apm read --json` 可还原上述分层，且测试覆盖。
- [ ] `npm test`、`npm run build` 通过。

## 风险与已确认决策

- **双排序队列**可能与用户直觉「同一批 chunk 为什么要出现两次」冲突——已在上方允许重复；若去重需单点策略并写测试。
- **中文分词**仍弱于英文 token：关键词抽取不足时，Primary 可能少于 3；允许为空或不足，但不得崩溃。
